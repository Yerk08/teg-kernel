Plan:
1. Бор для обработки путей внутри json данных.
	Преобразование данных из json в бор будет происходить в виде:
{Name:"cat", color:"dark", children:["AV","Va","av"]} ->

map <string, string>
/Name - "cat"
/color - "dark"
/children/0 - "AV"
/children/1 - "Va"
/children/2 - "av"

	При достижении конца вершин бора он должен начать искать опустошенные и заполнять их.
	Также добавить логарифмическую оптимизацию для поиска свободных точек.

	Требуется возможность преобразования строки в точку структуры бора и обратно
	Требуется возможность узнавать какие точки бора заняты и их количество для скриптов очистки
	Возможность сохранить префикс как число и использовать его для более быстрой обработки
	Возможность узнать все пути внутри данного префикса
	Возможность узнать все внутренние папки и файлы внутри данного префикса до следующего /
	Работать как RISC архитектура - узнать вершину бора и узнать значение в ней - две разные операции. Также позволять параллельно рыскать по бору, если это не меняет его структуры, а также изменение строки в вершине параллельно, если нету изменение одной и той же строки.

2. Ядро для запуска и взаимодействия со скриптами.
	При запуске атрибутами определяются количество выделяемых вершин в боре и путь к папке со скриптами.

	Параллельно держит все скрипты и дает им api через stdin и stdout. Запросы сводит в одну очередь, но выполняет запросы на чтение параллельно, а на запись нет и только после выполнения запросов на чтение.
	Каждый запрос - одна строка. Соответсвенно можно подавать сразу несколько запросов за одно обращение.

	Возможность отправки строки с одного скрипта на другой скрипт.
	Запрос обрабатывается с кодом, которое прислал скрипт и ответ также включает его.
	Обращение ко всем функциям бора.
	Возможность написания данных в stdout ядра и отправка новой строки stdin ядра каждому скрипту в виде "line <пришедшая строка>".

3. Документация и api взаимодействия.
	Скрипт представляет собой набор ассинхронных потоков, где один из них - функция чтения из консоли и записи всего в множество пришедших ответов. Все последующие потоки - отправляют запросы к ядру, а также иногда ждут от него ответ - они ожидают пока в множестве не появится новое значение.
	Если он хочет вызвать другой скрипт как функцию, то он использует код своего внутреннего счетчика, который он передает другому скрипту, и тот возвращает ответ с тем же кодом.

	Должен быть main скрипт который работает с консолью и принимает команды проекта.
	Так же он обрабатывает очередь событий которые он перенаправляет в порядке очереди на сторонние скрипты.
	Должен быть скрипт сохранения всех данных тем методом который нужен в проекте.

-- Необязательно --
	Ведется подраздел бора /tmp для сохранения всех временных данных между скриптами
	Должен быть скрипт очистки /tmp который удаляет старые записи, которые больше не используются.
